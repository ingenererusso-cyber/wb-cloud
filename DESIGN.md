# Vendra Agent Context

Этот файл — единый контекст для агентов и разработчиков.
Его цель: держать продукт, код, UX и визуальный язык в одной логике.

Если текущий код и этот документ расходятся, агент должен:
- сначала опираться на рабочий код и реальные пользовательские сценарии;
- затем предлагать правки так, чтобы проект двигался к правилам из этого файла;
- не делать “идеальный редизайн всего” без явного запроса.


## 1. Что это за продукт

Vendra — SaaS для продавцов Wildberries.
Фокус продукта сейчас:
- заказы и локализация;
- рекомендации к поставкам;
- пополнение по регионам;
- остатки FBS/FBO;
- карточки товаров и склейки;
- юнит-экономика;
- синхронизация данных и контроль качества данных;
- базовая подписка / триал / подтверждение email;
- поддержка через встроенный чат и форму обратной связи.

Тон продукта:
- не “маркетинговый шум”, а спокойный рабочий B2B-инструмент;
- помогает принимать решения, а не просто показывает таблицы;
- ощущается как дорогой, чистый, внимательный к деталям сервис.

К чему стремимся (каким я хочу видеть итоговый идеальный продукт):
- максимальная автоматизация/полуавтоматизация где возможно
- помощь в принятии решений для селлера по управлению магазином
- подсказки и напоминалки, по одному клику осуществляющие необходимые действия с разрешения селлера
- приложение, чтобы можно было получать уведомления на телефон и управлять продажами с телефона одним нажатием легко и без головной боли
- красивый, технологичный, современный (в плане используемых инструментов разработки) фронтэнд
- лаконичный и минималистичный бэкенд без хвостов и нерабочего кода
- сделать 2 тарифа: 1 - чтение (пользователь может пользоваться только READ методами и всемы вытекающими из этого данными и подсказками), 2 - чтение и запись (пользователь может пользоваться read and write API методами WB, может использовать все возможности по автоматизации, которые позволяют Write методы)
- сделать темную тему дизайна и переключатель в настройках


## 2. Текущая архитектурная реальность

Сейчас проект в основном server-rendered Django, а не SPA.

Основной стек:
- Django 6
- Django templates
- page-scoped CSS прямо внутри шаблонов
- fetch/XHR для тяжёлых или интерактивных частей
- SQLite локально для тестов, на сервере PostgreSQL
- отдельный `frontend/` с Tailwind/shadcn/Lucide уже существует, но не является основным UI-слоем проекта
- для графиков принят `Apache ECharts` как основной chart-слой вместо ручного SVG там, где нужен production-grade UI

Структура по смыслу:
- `core/` — модели, views, subscriptions, support, sync, большая часть продукта
- `core/services/` — отдельные сервисы по направлениям
- `app/services/supply_recommendations/` — расчёты supply recommendations
- `templates/` — основная UI-поверхность
- `wb_api/` — работа с внешним API
- `frontend/` — задел под более современный фронтенд

Текущий паттерн UI:
- страницы рендерятся Django-шаблонами;
- тяжёлые данные всё чаще выносятся в отдельные JSON API;
- страница открывается быстро, а данные догружаются после открытия;
- это уже сделано для нескольких сценариев и считается предпочтительным направлением.


## 3. Непереговорные продуктовые правила

Агент должен сохранять следующие принципы:

### 3.1. Безопасность данных и доверие
- Не показывать пользователю ложную “успешность”, если сервер не закончил действие.
- Любой долгий процесс должен иметь видимый статус.
- Любой сбой должен иметь дружелюбное сообщение для пользователя и технический лог для команды.

### 3.2. Продукт должен быть быстрым на вход
- Переход на страницу не должен ждать тяжёлых агрегаций, если их можно подтянуть после.
- Для тяжёлых таблиц, дашбордов, склеек, рекомендаций и карточек предпочтителен lazy load через API.
- Можно также реализовать колесо загрузки, где это необходимо

### 3.3. Данные должны быть объяснимыми
- Если метрика спорная или расчётная, рядом нужна подсказка или понятный лейбл.
- На экранах с бизнес-решением мы показываем максимум данных, чтобы расчет был прозрачным и пользователь мог увидеть ошибки, если они есть.

### 3.4. Сервис должен помогать действовать
- Отчёты должны подсказывать следующее действие.
- Если есть риск или проблема, лучше показать это как reminder / deck / warning на главной (сейчас реализуем это в процессе).


## 4. Текущие доменные сущности и как о них думать

Ключевые сущности:
- `SellerAccount` — основной контекст продавца
- `Order` — заказ WB
- `Product` — карточка товара
- `ProductCardSize` — связь размера/`chrt_id` с `nm_id`
- `SellerFbsStock` — FBS остатки
- `WarehouseStockDetailed` — FBO / складские остатки
- `SyncTask` — статус общей синхронизации
- `SignupLead` — регистрация до подтверждения email
- `UserSubscription` — триал / подписка / срок доступа
- `SupportThread`, `SupportMessage`, `TesterFeedback`, `AppErrorLog` — поддержка и диагностика

Термины, которые важно использовать последовательно:
- `nm_id` — карточка товара WB
- `imt_id` — склейка
- `vendor_code` / артикул продавца
- `FBO` и `FBS` нельзя смешивать в текстах и расчётах без явной подписи
- `локальные / нелокальные заказы` — важная бизнес-ось продукта
- `ДРР`, `ИЛ`, `локализация`, `коэффициенты приёмки`, `тарифы`, `пополнение`, `рекомендации к поставкам` — базовый словарь продукта


## 5. Текущие рабочие паттерны в коде

### 5.1. Views
- В `core/views.py` сейчас много логики. Это факт проекта, а не ошибка сама по себе.
- Новую тяжёлую бизнес-логику лучше выносить в helper/service, даже если endpoint остаётся в `views.py`.
- Если действие тяжёлое, у view должен быть лёгкий HTML-shell и отдельный JSON endpoint.

### 5.2. API-стиль
- Для JSON-ручек используем `JsonResponse`.
- Для валидации query params возвращаем `400`.
- Для внутренних ошибок:
  - логируем через `_log_app_error(...)`;
  - пользователю отдаём короткую понятную ошибку;
  - не светим traceback в UI.

### 5.3. Sync-поведение
- Общая синхронизация живёт вокруг `SyncTask`.
- UI уже умеет опрашивать текущий sync через верхнюю навигацию.
- Долгие процессы должны обновлять шаг/прогресс.
- Если процесс может зависнуть или устареть, должна быть явная стратегия stale-state.

### 5.4. Heavy pages
Текущий предпочтительный паттерн:
- быстрый HTML;
- затем fetch в API;
- состояния `loading / empty / error`;
- обновление URL для поиска/фильтров/страниц;
- отсутствие полной перезагрузки там, где это даёт реальную пользу.

Этот паттерн уже соответствует текущему направлению проекта для:
- рекомендаций к поставкам;
- пополнения по регионам;
- страницы склеек;
- части дашборда;
- reminder-действий;
- тяжёлых кусков карточки товара.


## 6. UX-правила для продукта

### 6.1. Основная навигация
- Верхнее меню — главный способ перемещения по системе. Он един для всех страниц сервиса.
- Секции продукта строятся вокруг бизнес-задач:
  - товары,
  - поставки и заказы,
  - аналитика,
  - поддержка,
  - аккаунт.

### 6.2. Состояния экранов
Каждый экран данных должен иметь:
- `loading`
- `empty`
- `error`
- нормальное содержимое

Если данных нет:
- не писать сухое “Нет данных” без контекста;
- объяснять, чего не хватает: синка, периода, остатков, карточек и т.д.

### 6.3. Действия пользователя
- Если действие неразрушающее и частое, его можно выполнять без reload.
- Если действие меняет состояние карточек/деков/фильтров — предпочтителен optimistic UI только если серверное действие надёжно.
- Если действие может быть опасным или медленным — лучше показать явное состояние выполнения.

### 6.4. Язык интерфейса
- Язык UI — русский.
- Формулировки короткие, спокойные, без канцелярита.
- B2B-лексика допустима, если она привычна продавцу WB.
- Системные тексты должны объяснять “что делать дальше”.


## 7. Визуальная система, исходя из текущего кода

Сейчас в проекте есть уже несколько визуальных направлений, но общий знаменатель такой:
- светлая тема по умолчанию;
- мягкие градиентные или воздушные фоны;
- белые/полупрозрачные карточки;
- радиусы 12–24 px;
- мягкие тени;
- зелёный как основной акцент (в особености для read части API);
- сине-голубые оттенки для части API, которая перезаписывает данные у WB (запись).

Базовые цвета, которые уже реально встречаются:
- зелёный акцент: `#14532d`, `#166534`, `#147a59`, `#065f46`
- тёмный текст: `#0f172a`, `#132033`, `#1f2a37`
- светлый фон: `#f8fafc`, `#edf3f7`, `#f4f6f8`

Что считать “нашим” стилем:
- premium SaaS
- clean
- data-driven
- visually calm
- slightly editorial, but not decorative for the sake of decoration

Что не надо делать:
- тёмную тему “по умолчанию” без явного решения
- кислотные акценты
- перегруженные таблицы без визуальной иерархии
- generic bootstrap-like UI
- “AI slop” блоки без характера


## 8. Правила по дизайну страниц

### 8.1. Дашборды
- Дашборд должен отвечать на вопрос “что происходит” и “что делать”.
- KPI-карточки, тренды и reminder-деки должны быть верхним уровнем.
- Лучше несколько сильных блоков, чем двадцать равнозначных.

### 8.2. Отчётные страницы
- Фильтры сверху.
- Далее summary, если он полезен.
- Далее таблица / список.
- Для тяжёлых отчётов допустим JS-render после загрузки JSON.

### 8.3. Карточные страницы
- Если объект сложный, разделять на “быструю” и “тяжёлую” части.
- Визуально важнее решение/вывод, чем сырой набор полей.

### 8.4. Reminder / deck UI
- Напоминания должны ощущаться как рабочие сигналы, а не как “маркетинговые баннеры”.
- Колода карточек — допустимый и уже принятый паттерн.
- Нижние карточки не должны конкурировать по кликабельности с верхней.


## 9. Правила по данным и расчётам

### 9.1. Расчёты должны быть детерминированы
- Явно задавать период.
- Явно разделять source-of-truth и derived values.
- Не смешивать FBO и FBS без явного комментария в коде и UI.

### 9.2. Если расчёт тяжёлый
Предпочтение:
1. snapshot после sync;
2. lazy API при открытии;
3. server render целиком — только если страница лёгкая.

### 9.3. Все пользовательские фильтры должны реально влиять на расчёт
- Не делать флаги “для вида”.
- Если есть тумблер вроде “Учитывать только FBS остатки”, он должен менять dataset, а не только текст.


## 10. Правила по ошибкам и диагностике

- Для серверных ошибок использовать `_log_app_error(...)`, если это пользовательский продуктовый сценарий.
- Пользовательское сообщение должно быть понятным и кратким.
- Если есть риск, что действие уже частично выполнилось, flow должен быть идемпотентным.

Это особенно важно для:
- регистрации,
- подтверждения email,
- синхронизаций,
- поддержки,
- изменения статусов/напоминаний,
- тяжёлых фоновых расчётов.


## 11. Правила по регистрации, доступу и оплате

Текущий flow:
- регистрация создаёт `SignupLead`;
- письмо с подтверждением отправляется через SMTP;
- переход по ссылке создаёт `User`, `SellerAccount`, `UserSubscription`;
- подписка сейчас стартует как триал;
- есть paywall и инициализация оплаты.

Что должен учитывать агент:
- confirm-flow должен быть идемпотентным;
- почтовый сценарий критичен для go-to-market;
- любые изменения в auth/signup/subscription должны сопровождаться тестами;
- нельзя ломать частично заполненные/исторически неровные схемы БД.


## 12. Правила по шаблонам, CSS и frontend

Текущая реальность:
- большинство страниц живут в Django templates;
- CSS часто лежит прямо в `<style>` внутри шаблона;
- общий top nav вынесен в partials;
- React/Tailwind слой пока вторичен.

Поэтому до отдельного решения:
- не нужно насильно переписывать существующие страницы в React;
- допустимо продолжать page-scoped CSS для изолированных экранов;
- новые UI-паттерны стоит делать так, чтобы потом их можно было вынести в общие partials или frontend-слой.

### 12.1. Правило по графикам

Новый стандарт проекта по графикам:
- не рисовать новые продуктовые графики вручную через inline `svg`, если для этого подходит `ECharts`;
- использовать локально подключенный vendor `ECharts`, а не CDN;
- общий reusable-слой для графиков держать в `static/js/vendra-charts.js`;
- графики должны иметь единый визуальный язык:
  - светлая подложка;
  - зелёный основной акцент;
  - мягкая area/fill-подложка там, где это уместно;
  - аккуратные tooltip;
  - авто-resize;
  - спокойные оси и grid без визуального шума.

Когда допустим ручной `svg`:
- только для очень простого одноразового мини-графика;
- или если библиотека объективно избыточна для конкретного кейса.

Предпочтительный порядок:
1. reusable helper в `static/js/vendra-charts.js`;
2. `ECharts`-конфиг на странице;
3. ручной `svg` только как исключение.

### 12.2. Правило по расчетным графикам

Если график строится не по историческому snapshot, а по восстановленной или теоретической модели:
- это нужно явно подписывать в UI;
- в API или коде должна быть отдельная заметка/лейбл, что расчет ориентировочный;
- пользователь не должен воспринимать такой график как точный бухгалтерский факт.

Пример правильного подхода:
- график затрат на хранение за прошлые 2 недели может восстанавливаться из текущего FBO-остатка и FBO-заказов;
- но рядом обязательно должна быть пометка, что это ориентировочная динамика для оценки, а не точный исторический остаток на каждый день.


## 13. Тестовая стратегия

Минимум для важных изменений:
- `manage.py check`
- точечные `TestCase` для новых API и критичных flow

Обязательно тестировать:
- auth/signup/confirm/subscription
- JSON API с фильтрами и датами
- тяжёлые отчёты, если меняется бизнес-логика
- сценарии без данных
- валидацию query params


## 14. Что считать “единым стилем” при дальнейшей чистке проекта

Если агенту нужно приводить проект к единому виду, по умолчанию двигаться сюда:

### Код
- меньше тяжёлой логики прямо в template/view;
- больше helpers/service-функций;
- JSON API для тяжёлых экранов;
- единый стиль ошибок и валидации.

### UX
- быстрый вход на экран;
- понятные статусы;
- меньше full reload;
- фильтры и диапазоны дат сверху;
- более явные действия пользователя.

### Визуал
- светлый premium dashboard;
- зелёный как основной бренд-акцент;
- крупные радиусы и мягкие тени;
- единая типографика и одинаковые паттерны карточек / таблиц / фильтров / empty states.

## 16. Правило для агента на будущее

Если задача неоднозначна, агент должен спрашивать себя в таком порядке:
1. Это помогает продавцу быстрее понять ситуацию и принять решение?
2. Это ускоряет экран или хотя бы не делает его тяжелее?
3. Это укладывается в текущую Django + fetch архитектуру без лишнего усложнения?
4. Это сохраняет светлый, премиальный, спокойный B2B-стиль?
5. Это не ломает существующие синки, регистрацию, подписку и продуктовые расчёты?

Если ответ “нет”, агент должен предлагать более простой и более согласованный вариант.
